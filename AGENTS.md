# Repository instructions

## Purpose

This repository develops a public reference implementation for portable enterprise knowledge used by AI systems and agents.

The repository must remain generic, synthetic, vendor-neutral, and safe for public release.

## Intellectual property boundaries

Do not include:

- Client names
- Real company data
- Private repository names
- Internal URLs
- Production schemas
- Proprietary ontology concepts
- Confidential workflow logic
- Credentials, secrets, tokens, or local file paths
- Content copied from private projects
- Claims that examples are production systems

All identifiers, entities, records, URLs, and organizations must be clearly synthetic.

Use `example.invalid` for hostnames.

## Git and authorship rules

- Do not create commits unless explicitly instructed.
- Do not push branches or tags unless explicitly instructed.
- Never add `Co-authored-by`, `Generated-by`, `Assisted-by`, or similar attribution trailers.
- Never list an AI model or automated tool as an author, committer, contributor, reviewer, or maintainer.
- Commit authorship must use the human operator's configured Git identity.
- Do not modify Git author or committer settings.
- Before proposing a commit, inspect the complete commit message and confirm that it contains no automated attribution.
- Do not add AI-use disclosures to repository files unless explicitly requested.

## Writing style

- Use direct, professional language.

- Avoid generic promotional phrasing, inflated claims, vague abstractions, and wording that sounds machine-generated.

- Before finalizing prose, check for local writing instructions if they are available.

## Markdown formatting

- Never hard-wrap prose paragraphs.
- Keep each paragraph on one physical source line.
- Use line breaks only where Markdown structure requires them.
- Preserve existing line breaks unless changing them is necessary.

## Working method

Before editing:

1. Inspect the repository.
2. State the proposed changes.
3. Identify IP or disclosure risks.
4. Wait for approval unless the current prompt explicitly authorizes implementation.

After editing:

1. Validate Markdown links.
2. Validate JSON and YAML.
3. Search for secrets, private paths, real organizations, and non-synthetic data.
4. Show changed files.
5. Do not commit.
